# RYQ AI Testing Framework — Tech Debt Register

## Phase 3.8 — No-Code Platform UI

**TD-001: Live stats not updating (stdout buffering)**
- **File:** `src/platform-server.ts`
- **Issue:** Live test stats in the Run Tests tab do not update in real time due to stdout buffering. The programmatic Playwright runner (introduced in Phase 4.8 storage wiring) partially addresses this — full fix requires the platform UI to read run status from DB via polling or SSE rather than stdout.
- **Impact:** Medium — testers see stale stats during a run. Results are correct after completion.
- **Resolution:** Implement DB-polling in the Run Tests tab. Platform server exposes a `/api/run-status/:runId` endpoint reading from `RunRepository`. Phase 5 programmatic runner makes this straightforward.
- **Status:** Open
- **Priority:** Medium

**TD-002: Platform runs headed locally**
- **File:** `src/platform-server.ts`
- **Issue:** When tests are triggered from the no-code platform UI locally, Playwright launches a headed browser. In most local dev scenarios this is acceptable but it is inconsistent with CI behavior.
- **Impact:** Low — cosmetic, no correctness issue.
- **Resolution:** Add a `PLAYWRIGHT_HEADED=false` env var check in the platform runner. Decision pending on whether headed local runs are desirable for debugging.
- **Status:** Open — decision pending
- **Priority:** Low

**TD-003: CI trend dashboard always shows 1 run**
- **File:** `src/dashboard-server.ts`
- **Issue:** The trend chart in the dashboard always shows a single data point when run in CI. Root cause is that `TrendRepository.computeAndUpsertForRun` aggregates by calendar day — multiple CI runs on the same day collapse to one trend record.
- **Impact:** Low — trends are correct over multiple days. Same-day multi-run visibility is lost.
- **Resolution:** Add an hourly granularity option to `TrendRepository`. Deferred to Phase 6 dashboard work.
- **Status:** Open — deferred Phase 6
- **Priority:** Low

**TD-004: Five unresolved generated tests from Phase 3.8**
- **File:** `src/tests/generated/`
- **Issue:** Five test cases generated by the Phase 3.8 Generate Test tab were not fully validated and remain in the generated/ directory without confirmed pass status.
- **Impact:** Low — they are not included in the main test run by default.
- **Resolution:** Review each generated test, fix or delete. Revisit once Phase 5 spec generator produces a reference output to compare against.
- **Status:** Open
- **Priority:** Low

## Phase 4.8 — Storage Foundation

**TD-005: SQLite lock directory artifact**
- **File:** `ryq-framework.db`
- **Issue:** A `.lock` directory is created by the SQLite POSIX lock mechanism. This is normal behavior but may appear as an unexpected artifact in the working directory.
- **Impact:** Cosmetic — no functional impact. `npm run db:migrate` must always run as a single process (which it does by design).
- **Resolution:** Add `ryq-framework.db-lock` to `.gitignore`. No code change needed.
- **Status:** Open
- **Priority:** Low

**TD-006: `app_name` hardcoded to `'saucedemo'` in migrated files**
- **Files:** `trend-analysis.ts`, `flaky-predictor.ts`, `release-notes.ts`, `notifier.ts`, `ai-triage.ts`, `adaptive-fixes.ts`, `gap-to-test.ts`, `knowledge-query.ts`, `dashboard-server.ts`, `platform-server.ts`, `query-server.ts`
- **Issue:** All repository calls passed `app_name: 'saucedemo'` as a hardcoded string. Multi-app support (Phase 5+) requires this to be dynamic.
- **Resolution:** Fixed in TD-006 cleanup sprint. `src/config/appConfig.ts` introduced. All 11 files updated to use `getAppName()`. APP_NAME env var defaults to `'saucedemo'`.
- **Status:** ✅ Resolved
- **Priority:** Medium — resolved before Phase 5.2

**TD-009: BFS crawler does not follow links beyond starting page**
- **File:** `src/onboarding/Crawler.ts`
- **Issue:** The crawler discovers only the starting page per role. Outbound links collected in `visitPage()` return empty on JavaScript-rendered apps because link elements may not be in the DOM at `networkidle`. BFS queue never grows beyond the initial URL.
- **Impact:** Medium — crawler cannot auto-discover pages. Hand-authored or config-seeded page lists required until fixed.
- **Resolution:** After `networkidle`, add explicit wait for anchor elements: `await page.waitForSelector('a[href]', { timeout: 5000 })` before harvesting outbound URLs. Also filter `href` values that are JavaScript void or hash-only links.
- **Status:** Open
- **Priority:** Medium — fix in Phase 5.2 patch before Phase 6
