# FORGE — Technical Debt Tracker

Single source of truth for open and resolved tech debt. `CLAUDE.md` points here
rather than duplicating this list — if you find a known-limitations note in
`CLAUDE.md` that isn't reflected here, that's a sign the two have drifted; update
this file and trim `CLAUDE.md` back down to a pointer.

When you find a new issue: add it here with the next available ID, don't fix it
inline unless it's trivial and unambiguously safe. Per the standing rule "design
before patching," anything structural gets a design conversation before code,
even if you could technically patch it in five minutes.

When you resolve one: move it to the Resolved table with the commit hash, don't
just delete the row — the history of what broke and how it got fixed is useful on
its own, especially for spotting repeat patterns (see "Common bug patterns" in
`CLAUDE.md`).

## Design decisions captured (not bugs, but constraints to respect)

These aren't open defects — they're explicit sequencing/architecture calls already
made, included here so they don't get silently violated by future work.

| Topic | Decision |
|---|---|
| Dashboard build order | Do not start Dashboard implementation while TD-014 (or any open verification/crawl correctness issue) remains unresolved. A drill-down dashboard built on unreliable underlying signal surfaces that unreliability as dashboard noise. Resolve foundation issues first. TD-013 resolved — no longer a blocker. |
| Dashboard data architecture | Dashboard must be a view layer on top of existing data (HealStore, flaky predictor scores, AI triage classifications, `run-history.json`/`trends.json`) — not a standalone reporting pipeline with its own data model. Avoids two sources of truth drifting apart. |
| TD-031 / OrangeHRM scope dependency | `Crawler.crawl()` always computes `stateGraph` for every app (`buildStateGraph()` runs unconditionally), but `detectFlows()`'s gating (`hasConfigFlows && isSpa`) skips `identifyCandidates()`/`enrichWithAi()` entirely for OrangeHRM today — meaning OrangeHRM cannot exercise the AI-inferred-flow path at all, regardless of how TD-031 is resolved. TD-031's fix should not be considered validated against OrangeHRM's actual production pipeline, and should not be claimed as "done" for app-agnostic use, until this gating is separately reviewed and revisited. (TD-031's own evidence bypassed this gating directly — calling `identifyCandidates()`/`enrichWithAi()` without going through `detectFlows()` — specifically because of this; see TD-031's Resolved entry.) |
| Verification confidence ≠ behavioral correctness (TD-033/TD-047, Option D) | `VerificationRunner`'s confidence scores indicate DOM-presence (or HTTP-status, for API apps) — not real-user-session behavior. Decided not to change `VerificationRunner`'s navigation pattern or check criteria (rejected for cost/complexity); instead redefined the contract and added a `verificationTier` field (`'dom-presence'` \| `'http-status'` \| `'interaction'`) to `ElementResult`/`FlowResult` so the limitation is visible in the report rather than implicit. Behavioral correctness (does this element actually work in a real user session) is the generated-spec CI run's job, not verification's. **Future features must not treat verification HIGH confidence as a predictor of generated-test pass/fail** — Dashboard's planned confidence-based gating, coverage scoring, or any similar feature must read `verificationTier` alongside `status`, not `status`/`confidenceLevel` alone. |
