# FORGE ‚Äî Technical Debt Tracker

Single source of truth for open and resolved tech debt. `CLAUDE.md` points here
rather than duplicating this list ‚Äî if you find a known-limitations note in
`CLAUDE.md` that isn't reflected here, that's a sign the two have drifted; update
this file and trim `CLAUDE.md` back down to a pointer.

When you find a new issue: add it here with the next available ID, don't fix it
inline unless it's trivial and unambiguously safe. Per the standing rule "design
before patching," anything structural gets a design conversation before code,
even if you could technically patch it in five minutes.

When you resolve one: move it to the Resolved table with the commit hash, don't
just delete the row ‚Äî the history of what broke and how it got fixed is useful on
its own, especially for spotting repeat patterns (see "Common bug patterns" in
`CLAUDE.md`).

## Resolved (Session 10 ‚Äî TD-018 root-cause fix)

| ID | Description | Resolved in | Notes |
|---|---|---|---|
| TD-018 | `PomGenerator.generatePage()`/`generateSmartProp()`/`generatePlainProp()` never deduplicated elements by `name` within a page, and the underlying `id` (`{pageId}:{name}`, the SmartLocator/HealStore key) collided too ‚Äî not just the generated property name. Root cause was upstream in `ElementClassifier`: `determineName()` derives names from DOM attributes shared across repeated structural elements (SauceDemo's 6 product cards all carry `data-test="inventory-item-name"` etc.), and `nameWithAi()`'s independent 20-element batches can't see each other's choices, so both paths can produce same-page collisions. | this session | Fixed at the root, not in `PomGenerator`'s output step. Added generic (no app-specific knowledge) repeated-container detection to `ElementClassifier.harvestElements()`'s `page.evaluate()`: walks an element's ancestor chain for the nearest node with 2+ siblings sharing its tag+class signature, recording `containerIndex` (position among siblings) and `containerHint` (first short text found inside that container instance) on `RawElement`. A new `deduplicateNames()` pass runs after AI naming and guarantees uniqueness via three tiers: (1) a `containerHint`-based descriptive suffix where the hint is itself unique within the collision group (`inventoryItemName` ‚Üí `inventoryItemNameSauceLabsBackpack`, ...); (2) a positional `containerIndex` suffix for members where the hint isn't unique or absent; (3) a running-counter safety net for any residual collision (fires for two unrelated repeated series that coincidentally share both a base name and a container index ‚Äî observed live on OrangeHRM, see below). `id` is rebuilt from the final name so the HealStore/SmartLocator key stays unique too. `PomGenerator.generatePage()` also gained a defensive `assertUniqueNames()` guard that throws rather than silently emitting invalid TypeScript if this invariant is ever violated by a future change. Verified live: re-crawled SauceDemo (zero duplicate names/ids across all 4 pages, confirmed programmatically) and regenerated its `generated/` directory ‚Äî `npm run check` now passes with zero errors (previously blocked by the duplicate-identifier compile error this entry describes). The stale `**/dashboard**` fixture bug noted in Session 9's scope check is also gone in the regenerated output (`fixtures.generated.ts` now correctly waits on `**/inventory.html**`). Also re-crawled OrangeHRM as a generalization check (not just SauceDemo) and found it had an **unlogged parallel instance of the same bug**, more severe: `viewEmployeeList` alone had 29 duplicate-name pairs (e.g. `employeeRow1Checkbox` x2), masked only because TD-019 separately blocks that app's full generation. Confirmed zero duplicate names/ids across all 21 OrangeHRM pages post-fix; tier 3's running-counter safety net fired for real there (OrangeHRM appears to render parallel desktop/mobile DOM copies of its tables, producing two independent repeated-row series with colliding base names *and* colliding container indices) ‚Äî direct live validation that the safety net is needed, not just theoretical. Restful Booker (`rest-api`) is unaffected ‚Äî `classifyPage()` short-circuits to the separate `classifyEndpoints()` static method before ever reaching DOM harvesting. **Found and fixed in the same change, not pre-existing:** the new `harvestElements()` helper functions initially used named `const`/`function` bindings, which broke immediately with `ReferenceError: __name is not defined` inside the browser ‚Äî `tsx` hardcodes esbuild's `keepNames: true`, which wraps any named function with a `__name(...)` call living at the top of the *Node*-side compiled module, not in the string `page.evaluate()` ships to the browser via `fn.toString()`. Fixed by binding each helper via array-destructuring (`const [signature] = [(el) => ...]`), which gets no inferred name in real JS and so nothing for esbuild to wrap ‚Äî confirmed empirically with an isolated repro before applying it. Logged as a new "common bug pattern" in `CLAUDE.md` since any future `page.evaluate()` helper in this codebase will hit it again. **Not fixed, explicitly out of scope (flagged, not silently absorbed):** regenerating SauceDemo's `generated/` directory for the first time also exposed TD-016 and TD-017 as **live, confirmed** blockers (previously only documented from static analysis) ‚Äî all 5 freshly-generated SauceDemo tests fail on a double-login pattern, not a TD-018 symptom. See updated TD-016/TD-017 entries above. |
